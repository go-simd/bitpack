# bitpack

[![CI](https://github.com/go-simd/bitpack/actions/workflows/ci.yml/badge.svg)](https://github.com/go-simd/bitpack/actions/workflows/ci.yml)
![coverage](https://img.shields.io/badge/coverage-100%25-brightgreen)

The **FastPFOR / simdcomp** bit-packing primitive in pure Go (CGO=0, stable Go,
no GOEXPERIMENT): pack blocks of **128 `uint32`** values, each using exactly
`bits` bits (1..32), into a tight little-endian bitstream â€” and the inverse.

```go
n := bitpack.Pack(dst, src, bits)   // src len a multiple of 128; n = 16*bits per block
bitpack.Unpack(out, dst, bits)
```

Output is **byte-identical to Lemire's simdcomp** (`SIMD_fastpackwithoutmask`):
the 128 integers are stored as 4 interleaved 32-int lanes. The bulk runs a
generated SIMD kernel (one per bit-width) on four architectures, with the scalar
reference everywhere else:

| arch | kernels | dispatch |
|---|---|---|
| **amd64** | SSE2 + AVX2 (`PSLLD`/`POR`/`PAND`, `VPSLLD`/â€¦) | `x/sys/cpu` (AVX2 when present, SSE2 baseline) |
| **arm64** | scalar | â€” |
| **ppc64le** | VSX (`LXVW4X`/`STXVW4X`, `VSLW`/`VSRW`, `VAND`/`VOR`) | build-tag (VSX is POWER8 baseline) |
| **s390x** | vector facility (`VL`/`VST`, `VESLF`/`VESRLF`, `VN`/`VO`, `VPERM`) | build-tag (vector facility is z13 baseline) |
| _others_ | scalar | â€” |

All kernels are generated by [go-asmgen](https://github.com/go-asmgen/asmgen).
Every path is byte-exact â€” verified by table tests, a scalar-equality test, and
`FuzzPack`/`FuzzUnpack`.

**Endianness.** The packed bitstream is always little-endian (to match simdcomp).
ppc64le is little-endian, so `LXVW4X`/`STXVW4X` load/store word elements
directly. s390x is **big-endian**: the `uint32` arrays are native big-endian (no
fix-up needed â€” the vector facility is also big-endian), and the only mismatch is
the little-endian packed stream, which the kernels byte-reverse per 32-bit word
with `VPERM` exactly at the packed-stream boundary (store on pack, load on
unpack). The byte-identical `FuzzPack`/`FuzzUnpack` targets pass under qemu on
s390x, confirming the big-endian port produces the same bytes as scalar/simdcomp.

## Test coverage

The Go code is at **100 % statement coverage**, gated in CI on the native amd64
and arm64 jobs and on the **ppc64le / s390x qemu jobs** (the build fails below
100 %). Both amd64 kernel paths â€” the AVX2 block-pair kernel and the SSE finisher
â€” are driven directly by the force test on the native AVX2 runner. The coverage
figure is of the Go code only â€” the generated `.s` SIMD kernels are not measured
by `go test -cover`; they are validated by differential tests against the scalar
reference plus the two fuzz targets, which run on every arch (under qemu for
ppc64le and s390x).

## Performance

Native amd64 (ubuntu-latest CI, AVX2, `-count=6`), throughput over the input
`uint32` stream, **vs the package's own scalar reference**:

| op | bits=8 | bits=16 | bits=24 |
|---|---|---|---|
| **Pack** SIMD | ~62.0 GB/s | ~57.5 GB/s | ~46.8 GB/s |
| Pack scalar | ~2.31 GB/s | ~1.92 GB/s | ~1.46 GB/s |
| **Unpack** SIMD | ~44.5 GB/s | ~42.5 GB/s | ~40.4 GB/s |
| Unpack scalar | ~2.08 GB/s | ~1.85 GB/s | ~1.38 GB/s |

That's **~21â€“32Ă—** over scalar â€” bit-packing is a textbook SIMD-friendly problem
(shifts + masks + ORs, no data-dependent control flow). go-asmgen generates the
per-width kernels.

The **ppc64le (VSX)** and **s390x (vector facility)** kernels are
**qemu-validated** (table tests + byte-identical `FuzzPack`/`FuzzUnpack` pass
under qemu); **native perf numbers are pending** (no native ppc64le/s390x runners
in CI).
